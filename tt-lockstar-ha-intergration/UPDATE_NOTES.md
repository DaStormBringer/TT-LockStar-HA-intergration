# Update and Hardware Validation Notes

> [!CAUTION]
> **VERY EXPERIMENTAL — NOT PRODUCTION READY.** These notes describe bounded tests on one physical lock. A successful test does not authorize unattended control or auto-unlock, and a software success response must not replace physical verification of the bolt.

This file contains the detailed release narrative and supervised hardware-test history that was previously embedded in the add-on README. See [README.md](README.md) for installation, current limitations, and the exact tested-compatibility scope. See [CHANGELOG.md](CHANGELOG.md) for the shorter release summary.

## Supervised validation history

Alpha.61 follows two read-only alpha.60 preparation failures. In legacy `direct` advertisement mode, the first request received a fresh M302 packet but its ESPHome GATT connection timed out; the next request then received no lock packet for 60 seconds, consistent with the documented single-subscriber contention between Home Assistant and a second direct ESPHome advertisement client. Switching the deployed option back to the supported shared Home Assistant feed immediately restored lock advertisements, but Home Assistant omitted `address_type`. The transport guessed static-random from the `DC` prefix even though the same physical lock's direct ESPHome packets had reported public address type `0`, and the resulting GATT attempt failed. Alpha.61 removes that invalid inference, lets the selected proxy use a source-specific type when available and otherwise the bridge's existing public fallback, and prevents a shared-feed advertisement from launching an unsolicited full metadata connection at startup. These changes do not retry or send lock/unlock; deployment still requires prepare-only validation before another supervised actuator test.

Alpha.60 exposes the already tested prepared-connection lease to Home Assistant as an MQTT-discovered `Prepare M302 Connection` button. One exact `PRESS` requests a 15-second read-only lease: it waits for a fresh advertisement, connects, verifies the TTLock command channel with a harmless time read, and keeps that channel responsive only until the lease expires. It does not call lock or unlock, it does not disconnect the lease early, and it does not grant permission to a later physical command. This enables a separate approach event to absorb wake and connection time while leaving actuator authorization independent.

The preceding alpha.59 direct lock test demonstrated why this is useful. The user confirmed the door closed and bolt locked after one command. The add-on logged exactly one actuator attempt, spent 7.716 seconds waiting for a fresh sleeping-lock advertisement and 3.860 seconds establishing a cache-miss GATT connection, then completed the authenticated lock phase in 610 milliseconds. Total command time was 12.189 seconds and no retry occurred. Alpha.60 does not change the actuator path; it moves only the read-only wake/connect work earlier when a meaningful approach sensor fires.

Alpha.59 follows the first physically successful alpha.58 prepared unlock. Preparation connected through the Craft ESPHome proxy from a fresh advertisement in 2.153 seconds using the saved GATT table, verified the TTLock command channel with an 83-millisecond read-only lock-time request, and kept the session responsive. The separately confirmed unlock reused that connection, completed its authenticated command phase in 659 milliseconds, and returned success on its first attempt. The user physically confirmed that the closed door was unlocked. Four keepalives occurred between the separate preparation and unlock client calls, adding roughly eight seconds of avoidable orchestration delay even though the actuator phase itself was sub-second.

The same live log exposed a safety issue: the high-level manager still labeled the successful command `attempt 1/2`. No second attempt occurred in that successful test, but a notification timeout or late disconnect could have caused an ambiguous physical command to be sent again without a fresh readiness check. Alpha.59 removes automatic actuator retries completely. Connection failure sends no lock/unlock payload; a false result, exception, timeout, or disconnect after a write clears recoverable local cache state, tears down the session, fails closed, and requires a new exact command-and-address confirmation before any later command. Read-only connection preparation and connection-level recovery remain available, but neither authorizes physical movement.

Alpha.58 follows the first alpha.57 prepared physical-command test. A 30-second read-only preparation lease waited 2.791 seconds for a fresh advertisement, connected through the Craft ESPHome proxy in 914 milliseconds, and reported ready. The separately confirmed unlock then claimed that same BLE session and the proxy accepted both command fragments, but the lock returned no notification response within five seconds. The command failed closed, no retry was sent, and the user physically confirmed that the bolt remained locked. This isolates the defect to the idle prepared-session handoff rather than advertisement discovery or GATT setup.

Alpha.58 verifies the subscribed command channel with a harmless lock-time read before reporting a preparation lease as ready. During the bounded 5-to-30-second lease it sends the same read-only request every two seconds, invalidates and disconnects immediately if a reply fails, and atomically stops and drains any in-flight keepalive before a separately authorized command claims the session. Preparation still cannot move the bolt or authorize a later action; every lock or unlock retains its own exact command-and-address confirmation. Fingerprint, PIN, card, passage-mode, and remote-unlock behavior are unchanged and remain outside this fix.

Alpha.57 then passed the live persistence test. One read-only `lock.features.get` completed in 3.71 seconds and returned 21 numeric feature identifiers, including auto-lock, audio management, PIN, IC card, fingerprint capability, passage mode, and remote-unlock configuration. The add-on reported auto-lock disabled (`0`) and sound on. After an add-on-only restart, the first local WebSocket status—while the lock was disconnected and before any new advertisement or Bluetooth request—restored the same capability flags, auto-lock `0`, and sound on from disk. No actuator, setting, credential, fingerprint, time-sync, reset, or firmware-update command was sent.

Alpha.57 follows the live alpha.56 persistence test. After the user woke the keypad, a single read-only retry completed the full metadata connection in 3.160 seconds and exposed auto-lock, PIN, card, fingerprint capability, and audio, with auto-lock disabled (`0`). The values remained present while the add-on process was running, but a deliberate add-on-only restart restored the pre-discovery values (`autoLockTime: -1` and every feature flag false). The SDK lock object's current snapshot contained the serialized feature list, while the ESPHome transport cache still held its pre-connection record because the late-disconnect path did not emit the SDK `dataUpdated` event. Alpha.57 replaces that matching transport record with the authoritative lock snapshot, synchronizes the cache, and then writes the same array to disk. The SDK snapshot also serializes and restores the auto-lock delay and last observed sound setting rather than reverting both to unknown defaults. No actuator, setting, credential, fingerprint, time-sync, reset, or firmware-update command was sent.

Alpha.56 follows the first live alpha.55 `lock.features.get` attempt. The Craft proxy connected from a 12-millisecond-old advertisement, the M302 completed feature decoding, and the live status immediately changed from every feature flag false to `hasAutoLock`, `hasPasscode`, `hasCard`, `hasFinger`, and `hasAudio` true with auto-lock correctly reported as disabled (`0`). The lock then disconnected during the SDK's final connection bookkeeping, so alpha.55 returned command failure before persisting the already complete feature `Set`. Alpha.56 accepts that narrow state as a successful read-only result while an absent or incomplete feature `Set` still fails closed. Its first restart test later proved that the save still used a stale ESPHome transport record; alpha.57 corrects that remaining persistence path. No actuator, setting, credential, fingerprint, time-sync, reset, or firmware-update command was sent.

Alpha.55 addresses the next non-biometric command blocker found after alpha.54 deployment. The live status correctly showed the M302 but reported every feature-gated capability as false because the pinned SDK did not serialize its discovered `featureList`; direct ESPHome mode deliberately defers the broad startup metadata read, and several manager methods checked capability flags before attempting any connection. Alpha.55 adds the read-only `lock.features.get` command, lazy one-time feature discovery for auto-lock, sound, PIN, card, passage-mode, and remote-unlock paths, and numeric feature-list persistence across restarts. It also preserves auto-lock zero instead of serializing that valid disabled value as unknown. Capability storage contains no credential or biometric values. Fingerprint enrollment and management remain last and are not exercised by this release.

Alpha.54 follows a manual physical-position comparison on the deployed alpha.53 add-on. With the door contact closed, the user confirmed the deadbolt physically locked; the next passive packet reported `isUnlock: false`, `hasEvents: false`, and `IDLE_NO_UNLOCK_SIGNAL`. The user then manually retracted the bolt without an electronic command and confirmed it physically unlocked; the next packet again reported the same values. The advertisement bit is therefore not a continuous bolt-position sensor on this M302.

To investigate whether any other advertisement byte changes, alpha.54 records a stable signature of the normalized manufacturer and service-data payload and retains up to 50 unique observations in process memory. Read-only `lock.status.get` returns the newest 20 observations with their raw bytes and timestamps; MQTT and WebSocket diagnostics expose the current signature. The history is cleared on restart, is never promoted to confirmed lock state, and must not be used by an automation. No actuator, authentication, setting, credential, time-sync, or firmware-update command is required to collect it.

Alpha.53 adds passive observability without broadening lock-state authority. The SDK-decoded advertisement `isUnlock` bit is now exposed separately through WebSocket status, read-only `lock.status.get`, and a Home Assistant MQTT diagnostic sensor. A set bit is labeled `UNLOCK_SIGNAL`; a clear bit is deliberately labeled `IDLE_NO_UNLOCK_SIGNAL`, not `LOCKED`, because it has not yet been proven to represent the physical bolt position on this M302 in every state. The observation is timestamped, marked `confirmed: false`, kept only in process memory, and never changes the confirmed lock entity. No actuator, authentication, setting, credential, time-sync, or firmware-update command is required to collect it.

After deployment, the dedicated Craft proxy immediately delivered the M302 advertisement at approximately -73 to -64 dBm. The new Home Assistant entity `sensor.front_door_lock_m302_2f9485_advertised_lock_state` reported `IDLE_NO_UNLOCK_SIGNAL` with `advertised_is_unlock: false`, `advertised_has_events: false`, and `advertised_confirmed: false`. The same read-only API response independently reported confirmed deadbolt state `UNKNOWN` and magnetic contact `CLOSED`. This proves the passive observation is reaching MQTT, WebSocket status, and `lock.status.get` without being promoted to confirmed `LOCKED`. No lock, unlock, authentication, setting, credential, time-sync, or firmware-update command was sent during validation.

Alpha.52 follows alpha.51's first deployed read-only preparation request. The add-on and Craft proxy were healthy and continued logging M302 packets, but the command's strict one-second freshness gate observed no qualifying packet inside the normal 15-second physical-command window. Preparation failed closed with `Command failed: lock.connection.prepare`; it never opened BLE and sent no authentication, lock, unlock, settings, credential, or firmware-update payload. Because preparation is meant to begin in the background before arrival, alpha.52 gives only this read-only command up to 60 seconds to observe the sleeping lock. Its 5–30 second reusable lease still begins only after connection succeeds. Physical lock/unlock timing and retry limits are unchanged.

Alpha.51 targets the delay isolated by the physically confirmed alpha.50 unlock. The command returned success and the user confirmed that the bolt retracted, but the 7.287-second total contained a 5.584-second wait for the sleeping M302's next advertisement. Once that packet arrived, the cached BLE connection completed in 1.100 seconds and the authenticated unlock phase completed in 601 milliseconds. The remaining first-contact delay is therefore the lock's sleep/advertisement cadence rather than GATT discovery or the actuator protocol.

The new read-only `lock.connection.prepare` command opens the same command-only BLE/GATT session without sending lock, unlock, authentication, settings, time-sync, credential, or firmware-update commands. It holds the session for 15 seconds by default, accepts only 5 through 30 seconds, automatically disconnects when unused, and lets the next command claim the existing mutex/session. The following physical command still performs TTLock authentication and requires the exact command-and-address confirmation. This is intended for an earlier local presence event to absorb the sleep delay before a separately authorized action; it is not authorization for unattended auto-unlock. Live validation must also check battery impact and whether the brief connection interferes with the TTLock phone app or G2 gateway.

Alpha.50 was deployed with the configured Craft Everything Presence Lite proxy and preserved every add-on option. Two read-only firmware queries returned **6.4.43.24052101**. The first uncached connection completed in 1.181 seconds. An immediate cached reconnect timed out at the new 4.0-second boundary, preserved its GATT table, and recovered uncached; setup still took 14.908 seconds. After a cooldown, a third read used the preserved GATT cache and connected in 999 milliseconds. This validates bounded recovery and cache preservation, while also showing that an immediate post-disconnect reconnect can remain slow.

Alpha.49 is an observability and API-correlation release. Generic `capabilities` and `command` requests can carry an optional top-level string or numeric `requestId`; the matching direct response or error echoes it, while broadcast status messages remain uncorrelated. Connection logs now separate per-lock mutex wait, fresh-advertisement wait, BLE connection, authenticated command, and total operation timing. Advertisement freshness, retries, authentication, command payloads, response validation, and disconnect behavior are unchanged. The built `amd64` image passes 85 JavaScript tests and 7 ESPHome bridge tests and reports SDK 0.3.34 with 40 commands. This automated validation does not require a physical actuator command; live timing remains separately supervised.

Alpha.48 exposes the pinned JavaScript SDK's supported high-level command surface through a capability-discoverable generic WebSocket API. Exact command-name and lock-address confirmation is mandatory for actuator, security-sensitive, and destructive commands. Protected SDK internals, raw command injection, firmware update, and the M302-inappropriate bicycle status query remain unavailable. Existing frontend message types are unchanged.

Alpha.48 also reuses a successfully discovered ESPHome GATT service table and MTU across short-lived lock sessions. A cached failure clears the local and ESPHome cache before one uncached retry. This removes repeat service discovery only; it does not cache or bypass TTLock authentication, command challenges, notification setup, response validation, or physical verification. Automated coverage passes.

After deployment, the generic `lock.device_info.get` command returned firmware **6.4.43.24052101** from the physical M302 and repeated sessions logged GATT cache hits. A subsequent supervised `lock.unlock` command used a cached GATT connection, completed the BLE connection in 2.019 seconds, completed the authenticated command phase in 624 milliseconds, and returned success in 7.853 seconds total. The user physically confirmed that the bolt retracted. This single alpha.48 sample is 279 milliseconds slower than the alpha.47 unlock baseline, so it validates the cache path and authenticated actuator operation but does not demonstrate a latency improvement.

The deployed alpha.47 baseline immediately before this work completed a sleeping-keypad round trip through the dedicated Craft proxy. Unlock connected in 1.221 seconds and returned authenticated success in 7.574 seconds total; the user confirmed the bolt retracted. Lock connected in 3.769 seconds and returned authenticated success in 5.569 seconds total; the user confirmed the bolt extended. These alpha.47 measurements are the comparison baseline for alpha.48, not proof that alpha.48 is faster.

Alpha.47 follows a direct raw-advertisement diagnostic with the Craft proxy on Everything Presence Lite 1.5.0 / ESPHome 2026.5.2. After Home Assistant released the proxy's single advertisement subscription, Craft connected to the M302 in 3.567 seconds at MTU 23, completed the general-data GATT writes, and disconnected normally. That automatic startup metadata refresh consumed the lock's first connectable wake; the immediately following read-only firmware session saw fresh advertisements but timed out connecting. Alpha.47 therefore defers only the automatic discovery refresh in direct ESPHome mode so the first connection is reserved for an explicit request. Shared Home Assistant advertisement behavior and all actuator commands remain unchanged.

After alpha.47 was deployed, direct discovery registered the paired M302 and logged that automatic metadata refresh was deferred. An explicitly armed firmware request then accepted a 24 ms-old keypad advertisement, opened the command-only route, and returned firmware **6.4.43.24052101** with `command: "COMM_READ_DEVICE_INFO"`, `infoType: "FIRMWARE_REVISION"`, and `readOnly: true`. The add-on disconnected afterward. This validates only the read-only firmware query on this one M302 and does not broaden actuator or unattended-operation claims.

Alpha.46 follows the first reserved alpha.45 live firmware attempt. The Craft/door proxy received the M302 wake at approximately -66 dBm with all three connection slots free, and the firmware route began a command-only connection from a 50 ms-old advertisement. GATT connections through Craft, Living Room, and Master Bedroom all timed out. The shared Home Assistant advertisement feed omits BLE `address_type`; alpha.45 therefore supplied public address type `0`, unlike the previously successful direct ESPHome advertisement path. Alpha.46 preserves explicit ESPHome values and infers static-random type `1` for a missing address with the required high-bit pattern. It also gives the bridge enough time to return a successful second-proxy candidate rather than timing out the JavaScript caller while Python continues in the background. At the end of alpha.46, the read-only firmware response was still pending; no actuator, settings, time-sync, or firmware-update command was sent.

Alpha.45 makes the firmware request explicitly armable. It reserves the paired M302 from the failed-startup metadata queue, waits up to 60 seconds for a fresh keypad advertisement, then uses the existing command-only `COMM_READ_DEVICE_INFO` / `FIRMWARE_REVISION` path. This fixes the alpha.44 live-test race in which the startup full-data refresh consumed the wake before the firmware request began. No actuator, settings, time-sync, or firmware-update command is added.

Alpha.44 exposes a bounded read-only firmware query over the add-on WebSocket API. A request shaped as `{"type":"firmware","data":{"address":"DC:47:11:85:94:2F"}}` opens a command-only BLE session, sends TTLock `COMM_READ_DEVICE_INFO` for `FIRMWARE_REVISION`, returns a `firmware` response, and disconnects. It does not send lock, unlock, settings, time-sync, or firmware-update commands. The implementation is unit tested but remains pending a live response from the tested M302.

Alpha.43 changes only the advertisement ownership path. Home Assistant remains the ESPHome advertisement subscriber and publishes the supported `bluetooth/subscribe_advertisements` WebSocket stream to TT LockStar through the Supervisor-authenticated Core proxy. Separate ESPHome native API clients retain active scan mode and GATT operations. Home Assistant scanner source addresses are mapped back to configured proxies before the lock advertisement reaches the freshness gate, preserving strongest-recent-proxy selection. The legacy direct subscription remains an explicit diagnostic option and is never selected automatically. This change is awaiting a deployed read-only firmware/version request and does not authorize a lock or unlock command.

On 2026-07-12, `0.1.0-alpha.13` completed one supervised physical unlock and lock cycle on an M302 lock through a direct `hci0` adapter. Both commands returned success on their first command-only attempt, the deadbolt movement was confirmed at the door, and Home Assistant changed from `unlocked` to `locked` accordingly. Signal during the test was approximately -83 to -79 dB.

This is a single-device experimental result, not production qualification. Noble still emitted an intermittent `unknown peripheral null connected` warning during the successful lock attempt, so unattended control and auto-unlock remain out of scope.

Alpha.14 replaced the former `@abandonware/noble` raw-HCI runtime with `@stoprocent/noble` 2.5.5 using BlueZ over host D-Bus. Two supervised unlock tests failed: the lock was visible at approximately -77 dB, but the BLE session disconnected during generic characteristic reads or while waiting for the `checkUserTime` response. The second test used an open door, extended deadbolt, and awake keypad, so strike resistance was excluded.

Alpha.15 invalidates stale D-Bus `Connected` and `ServicesResolved` cache entries after a remote disconnect and gives command-only connections a shorter setup path that discovers the TTLock service without rereading cached generic device information. The image must still pass another supervised open-door hardware test before this change can be considered validated.

The installed alpha.16 image completed a supervised open-door physical cycle. Unlock returned `true` on the first manager attempt, the bolt retracted, and Home Assistant changed to `unlocked`. An immediate relock then failed: attempt 1 hit the 55-second hard timeout and attempts 2–3 could not connect, leaving the bolt physically retracted. Restarting only this add-on cleared Noble's stale raw-HCI session; the following lock returned `true` on its first manager attempt, extended the bolt, and changed Home Assistant to `locked`. Raw HCI can operate the lock, but sequential command recovery remains a blocking reliability defect.

Alpha.17 replaced the inherited 40–55 second command connection waits with a bounded command-only policy. In supervised testing, one lock command completed physically in about 18 seconds. The following unlock exposed a stale optimistic state but did not move the bolt: the first connection reached the 12-second outer timeout, the retry could not connect, and the user confirmed the door remained locked. The timeout bounded the failure, but the expired setup continued in the background and contaminated the retry.

Alpha.18 explicitly cancelled and reset an expired connection before retrying, and limited command discovery to TTLock service `1910`. In supervised testing, an unlock still failed safely: the first attempt reached its 12-second timeout, the retry collided with Noble's still-pending HCI cancellation and reported `unknown peripheral null connected`, and the user confirmed the physical deadbolt remained locked.

Alpha.19 waits for Noble's pending connection slot to drain before retrying. If cancellation does not drain within its bounded cleanup window, it fails closed without issuing the known-stale retry. Elapsed-time logging is included for the next supervised test.

Alpha.28 retained the native `bluez` option that bypasses Noble and implements discovery, connection, GATT enumeration, notifications, reads, and writes directly through BlueZ D-Bus. Native disconnect removes only the host-unpaired target cache so the next wake creates a fresh device object.

Alpha.29 records the first physically verified native BlueZ round trip on the front-door lock without restarting the add-on. Unlock returned its authenticated success response in 4.32 seconds and the user confirmed that the bolt retracted. The immediate lock's first connection timed out before the lock command was written; its bounded retry connected in 447 ms, returned the authenticated lock response in 9.17 seconds total, and the user confirmed that the bolt extended. The native process loaded no Noble runtime modules. New installations still default to legacy raw HCI, and one supervised cycle is not enough to authorize unattended or automatic lock control.

Alpha.30 targets the remaining command latency. Native BlueZ keeps discovery active while initiating a command connection, retries after 100 ms instead of 1.5 seconds, and bounds a failed command connection at 4.5 seconds instead of 6 seconds. It preserves authenticated command handling, the two-attempt limit, and every non-native transport policy. These timing changes require supervised physical validation before their performance or reliability is established.

Alpha.31 adds a fully local `esphome_proxy` transport. A small Python bridge uses the ESPHome native API for advertisements, proxy selection, active BLE connections, GATT enumeration, reads, writes, and notifications while the existing JavaScript SDK continues to own TTLock credentials, encryption, and commands. It does not use TTLock Cloud or the G2 gateway. This path must pass read-only discovery and supervised physical command testing before it is considered hardware validated.

Alpha.32 follows the first deployed read-only test, where an M302 keypad wake was not observed after the add-on reclaimed the proxy subscriptions. It explicitly requests active scanning on every proxy connection and documents that ESPHome's newest Bluetooth API subscriber owns the proxy stream. This remains a discovery fix awaiting the next live wake test; it is not a successful lock-operation claim.

Alpha.33 addresses the next deployed wake test: active scanning was requested successfully, but the paired M302 still did not reach the TTLock matcher. ESPHome advertisements from the exact stored paired-lock MAC are now accepted even when service UUID `1910` is absent; UUID filtering remains mandatory for unknown addresses. This is awaiting another live wake test and is not a lock-operation claim.

Alpha.34 follows a synchronized sole-subscriber capture. The deployed ESPHome firmware rejected a second Bluetooth subscriber, so the add-on was stopped for the bounded diagnostic. The Living Room proxy then received five M302 wake packets at approximately -81 to -79 dBm, while the Master Bedroom proxy delivered no packets. The packets arrived through ESPHome's raw-advertisement batch message and contained the paired MAC, UUID `1910`, M302 name, manufacturer data, and service data. Alpha.34 consumes that raw format and decodes it locally. No lock command was sent, and this remains awaiting a deployed metadata session.

After a supported integration handoff made the Living Room proxy the dedicated TT LockStar Bluetooth subscriber, alpha.34 completed a full read-only metadata session in 18.9 seconds and updated lock time through MQTT. Its first supervised unlock test then failed safely: both ESPHome connection attempts timed out before any unlock payload was written, and the user confirmed the deadbolt remained locked. Alpha.35 widens only the ESPHome connection window and allows its failed-connect cleanup to finish before a retry; it does not change command authentication or any direct-adapter timing.

Alpha.35 then completed the first physically confirmed ESPHome proxy unlock on this M302. It connected through the Living Room proxy and returned authenticated success in 2.67 seconds; the user confirmed the deadbolt retracted. Two supervised relock commands subsequently connected but lost the BLE session during the authenticated `checkUserTime` exchange, before either actuator payload was written; the user confirmed the bolt remained unlocked. Alpha.36 sends the three BLE fragments of each handshake as one ordered bridge transaction to reduce process/API round-trip latency while preserving 20 ms pacing and the full TTLock authentication exchange. It still requires supervised hardware validation.

Alpha.36 then completed a physically confirmed ESPHome proxy lock on its first attempt in 4.66 seconds. The authenticated response returned and the user confirmed the deadbolt extended. Three subsequent unlock tests failed safely: the immediate attempt could not establish another GATT session, and later attempts connected but disconnected during `checkUserTime`; no unlock actuator command was written and the user confirmed the bolt remained locked each time. The official TTLock Android flow uses administrator authentication for an administrator unlock, while the inherited JavaScript SDK always used the ordinary-user time check. Alpha.37 selects `CHECK_ADMIN` only when an imported administrator password is present and retains the existing time-window path for ordinary eKeys. This remains supervised-test-only.

Alpha.37 then completed a physically confirmed administrator-authenticated unlock through the Living Room ESPHome proxy on its first attempt in 3.60 seconds. The following relock still used the inherited ordinary-user `checkUserTime` path, disconnected before the actuator payload, and timed out on its bounded retry; the user confirmed the bolt remained unlocked. TTLock's V2 interface exposes a corresponding `lockByAdministrator` operation. Alpha.38 therefore selects `CHECK_ADMIN` for lock as well as unlock when administrator credentials are present, while retaining `CHECK_USER_TIME` for ordinary eKeys. The lock command payload and response validation are unchanged, and this remains supervised-test-only pending live validation.

Alpha.38 did not validate that administrator-lock assumption on this M302. Both bounded attempts connected through the Living Room ESPHome proxy, sent the administrator-check fragments, and disconnected while waiting for the response before any lock actuator payload was sent; the user confirmed the bolt remained unlocked. The official React Native wrapper exposes a generic `controlLock(LOCK, lockData)` boundary and leaves authorization selection inside TTLock's native SDK, while the pinned JavaScript SDK analysis documents administrator authentication for unlock but the user-time check for lock. Alpha.39 therefore reverts only lock to `CHECK_USER_TIME`; the physically validated administrator-unlock path remains unchanged. This is a compatibility rollback awaiting another supervised test, not a production-reliability claim.

Alpha.39 confirmed the rollback but did not complete a lock operation: both attempts connected through the Living Room proxy in under 1.4 seconds, then disconnected while waiting for the `CHECK_USER_TIME` response, and the user confirmed the bolt remained unlocked. A comparison lock from the TTLock phone app succeeded; the ESPHome advertisements changed from `newEvents: true` to `lockedStatus: true`, demonstrating that the lock actuator and passive proxy observation path were healthy. ESPHome write-without-response calls confirm local queueing rather than peripheral consumption, so alpha.40 increased only the inter-fragment drain interval from 20 ms to 100 ms.

Alpha.40 started successfully and its metadata session completed multipart commands with 100 ms pacing, but the supervised unlock failed safely: both bounded attempts connected and sent the two-fragment administrator check, then disconnected while waiting for its response; the user confirmed the bolt remained locked. The bridge was retaining aioesphomeapi's notification-subscription marker after the underlying BLE connection ended. Because ESPHome notification registration belongs to the current connection, a reconnect could falsely report that it was already subscribed and omit the new start-notify request. Alpha.41 removes the stale local callback on every observed disconnect and defensively before reconnecting, forcing each BLE session to establish its own notification path. The command authentication, payloads, response checks, 100 ms fragment pacing, and two-attempt fail-closed limit are unchanged. This fix remains awaiting supervised validation.

Alpha.41 then passed three consecutive supervised operations through the Living Room ESPHome proxy. Administrator unlock completed on attempt one in 6.815 seconds, the following user-time-authenticated lock completed on attempt one in 1.722 seconds, and the next administrator unlock completed on attempt one in 4.262 seconds. The user physically confirmed every bolt movement, and the bridge cleared the connection-scoped notification callback after each operation. A lock test without touching the keypad exposed a separate latency defect: attempt one started from a 9.987-second-old advertisement and spent 20 seconds timing out; attempt two used a 50 ms-old advertisement, connected in 8.363 seconds, sent the authenticated lock exchange, then lost the final response. The command failed after 41.844 seconds and the user confirmed that the bolt remained unlocked.

Alpha.42 requires an ESPHome advertisement no older than one second before starting a command connection. It also honors the SDK's eight-second command-only connection timeout instead of silently extending it to 18 seconds; longer full-metadata sessions retain their existing timeout. This prevents an obsolete connection attempt from consuming the lock's usable BLE command window and lets the existing bounded retry start sooner. Authentication, command payloads, response validation, notification cleanup, 100 ms fragment pacing, and the two-attempt limit are unchanged.

Deployed alpha.42 completed an administrator unlock without a keypad wake on attempt one in 6.281 seconds, and the user physically confirmed that the bolt retracted. The immediate no-wake lock then failed both Living Room proxy connection attempts after 31.119 seconds; the user confirmed that the bolt remained unlocked. The add-on was subsequently configured with both the Living Room proxy (`192.168.1.30`) and Master Bedroom proxy (`192.168.1.55`). A redundant-proxy no-wake lock returned authenticated software success after 26.472 seconds: Master Bedroom timed out first, then Living Room connected and completed the command. The user ended testing before reporting the physical bolt position, so that final operation is explicitly **not** counted as physically validated.
